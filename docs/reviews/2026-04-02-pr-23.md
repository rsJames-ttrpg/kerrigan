## Blocking Issues

### 1. `polars` declared but never used (blocks Buck2 build)

**Files:** `src/evolution/Cargo.toml`, `src/evolution/BUCK`

The commit message for the metrics module says "Polars-backed metrics aggregation" but `metrics.rs` uses only standard `HashMap` and `Vec` operations â€” not a single `use polars::` statement exists anywhere in the crate. The `polars` dependency (v0.46) is declared but unused.

**Impact:**
- `third-party/BUCK` has not been updated with polars (acknowledged in the PR), which breaks the Buck2 build entirely.
- Polars is a large crate (~72MB compiled). This severely impacts build times on the Raspberry Pi target hardware.

**Fix:** Remove `polars` from `src/evolution/Cargo.toml` and `src/evolution/BUCK`. Since all other new dependencies (`flate2`, `chrono`, `serde`, `serde_json`, `anyhow`, `tracing`) are already present in `third-party/BUCK` (used by other crates), removing polars means `buckify.sh` does not need to be run at all and the Buck2 build will succeed.

### 2. Evolution actor count-trigger fires continuously in production

**File:** `src/queen/src/actors/evolution.rs`, lines 36â€“47

```rust
let runs = match client.list_runs(Some("completed")).await {
    Ok(r) => r,
    Err(e) => { ... continue; }
};
completed_since_last = runs.len(); // simplified: count all completed
```

After the first analysis completes, `completed_since_last` is reset to 0 (line 94). On the very next poll (60 seconds later), `list_runs(Some("completed"))` returns **all** completed runs ever â€” not just those since the last analysis. If the system has â‰Ą10 completed runs (the default `run_interval` threshold), the count trigger fires immediately and permanently on every poll cycle.

**Impact:** In a mature system this creates a continuous flood of `evolve-from-analysis` job runs, each spawning a drone to create GitHub issues. This would generate significant API cost and noise.

**Fix:** Track completed run count as a delta since `last_analysis_time`. Options:
1. Use the existing `since` artifact filter approach: count artifacts created after `last_analysis_time` as a proxy for completed runs.
2. Add a `since` filter to the `list_runs` endpoint.
3. Persist the count of completed runs at last-analysis time and subtract it each poll.

## Non-Blocking Issues

### 3. `repo_run_interval` config field is dead code

**File:** `src/queen/src/config.rs`, line 179-180

`EvolutionConfig` defines `repo_run_interval: usize` with a default value of 5, but the evolution actor only implements global scope and never reads this field. This creates confusion about whether per-repo analysis is supported.

**Fix:** Remove the field (v1 doesn't support per-repo scope) or add a comment like `// reserved for v2 per-repo analysis`.

---

### 4. No validation of `EvolutionConfig` fields at startup

**File:** `src/queen/src/config.rs`, `Config::validate()`

`EvolutionConfig` fields are not validated:
- `min_sessions = 0` would cause analysis to always run (passes the `>= 0` check silently).
- An invalid `time_interval` string (e.g., `"24z"`) causes `parse_duration` to fail at runtime, but the `unwrap_or(Duration::from_secs(86400))` in the actor silently falls back with no warning logged.

**Fix:** Add validation for `min_sessions > 0` and validate `time_interval` format during `Config::validate()` (or at minimum at actor startup with an explicit log warning).

---

### 5. Silent failure when report JSON serialization fails

**File:** `src/queen/src/actors/evolution.rs`, line 71

```rust
let report_json = serde_json::to_string_pretty(&report).unwrap_or_default();
```

If serialization fails (unlikely given the types, but possible if serde hits a depth limit or NaN float), `report_json` silently becomes an empty string and the drone receives an empty JSON block in its task.

**Fix:** Handle the error explicitly â€” log a warning and skip submitting the job:
```rust
let report_json = match serde_json::to_string_pretty(&report) {
    Ok(j) => j,
    Err(e) => { tracing::warn!(error = %e, "failed to serialize evolution report"); continue; }
};
```

### 6. Fragile compression event detection

**File:** `src/evolution/src/parse.rs`, line 124

```rust
if msg.contains("compress") || msg.contains("truncat") || msg.contains("summary of the conversation") {
    compression_events += 1;
}
```

Any system message containing these common English words (e.g., a task saying "compress the image files" or "truncate the log output") would be counted as a context compression event, producing false positives in the metrics.

**Fix:** Use more specific Claude Code patterns, or add a `// NOTE: heuristic approximation` comment and document the false-positive risk.

### 7. Non-deterministic `extract_first_arg` for retry detection

**File:** `src/evolution/src/parse.rs`, lines 141â€“152

```rust
for (_key, val) in obj {
    if let Some(s) = val.as_str() {
        return s.chars().take(100).collect();
    }
}
```

While `serde_json::Map` preserves insertion order, Claude Code's output JSON field ordering may not be stable across versions. The retry detection logic in `detect_retries` relies on consistent `first_arg` values to identify same-target retries.

**Fix:** Use an explicit priority lookup for known key names:
```rust
for key in &["file_path", "pattern", "command", "path"] {
    if let Some(s) = obj.get(*key).and_then(|v| v.as_str()) {
        return s.chars().take(100).collect();
    }
}
```

### 8. Median calculation uses wrong index for even-length slices

**File:** `src/evolution/src/metrics.rs`, line 127

```rust
sorted[sorted.len() / 2]
```

For even-length slices (e.g., 4 elements), this returns element at index 2 (upper-middle) rather than the average of elements at indices 1 and 2. The statistical inaccuracy is minor but `median_turns` is a reported metric.

### 10. RFC3339 `since` timestamp not URL-encoded

**File:** `src/nydus/src/client.rs`, lines 419â€“421

```rust
params.push(format!("since={s}"));
```

The `since` string is interpolated directly into the query string without URL encoding. RFC3339 timestamps with non-UTC timezone offsets (e.g., `+05:30`) contain `+` signs which HTTP servers interpret as spaces. **Low risk in practice** because `Utc::now().to_rfc3339()` always produces a `Z`-suffixed string, but the client code is not defensively correct.
