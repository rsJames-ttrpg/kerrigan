## Summary

This PR adds `creep-cli` (a thin gRPC client crate for Creep's FileIndex service), a `creep-discovery` Claude Code skill plugin, drone lifecycle hooks to register/unregister workspaces, and container build integration. The implementation is clean and follows established project conventions. There are **two bugs** and **one significant functional gap** that should be addressed before merge.

---

## Bugs

### 1. JSON injection in `cmd_register` and `cmd_unregister` (MEDIUM)

**File:** `src/creep-cli/src/main.rs:156,176`

The JSON output uses manual string interpolation rather than a serializer:

```rust
// cmd_register
println!(r#"{{"files_indexed":{count},"path":"{path}"}}"#);

// cmd_unregister
println!(r#"{{"unregistered":true,"path":"{path}"}}"#);
```

A workspace path containing a `"` or `\` character will produce invalid JSON (e.g. a path like `/tmp/path with "quotes"` breaks the output). All other commands use `print_json()` with `serde_json`. These two should too.

**Fix:** Use `serde_json::json!()` or a small anonymous struct.

---

### 2. Creep is never started in the container entrypoint (HIGH — functional gap)

**File:** `deploy/dev/entrypoint.sh`

The entrypoint starts `overseer` and `queen` but not `creep`. The drone hooks call `creep-cli register <workspace>` during `setup()`, but since Creep isn't running, the call will fail. The failure is handled gracefully (warn + continue), so it won't break anything — but the feature effectively doesn't work in the container.

`creep` is built and staged (`deploy/dev/.stage/bin/creep`) and copied to `/opt/kerrigan/bin/creep` in the Dockerfile, so the binary is available. It just needs to be started.

**Suggested addition to `entrypoint.sh`** (before starting Queen):
```bash
echo "=== starting creep ==="
/opt/kerrigan/bin/creep &
CREEP_PID=$!
sleep 1  # brief wait; Creep starts fast
if ! kill -0 "$CREEP_PID" 2>/dev/null; then
    echo "WARNING: creep exited unexpectedly — file discovery will be unavailable"
fi
```

## Minor Issues

### 3. Proto duplication — no sync mechanism

**Files:** `src/creep-cli/proto/creep.proto`, `src/creep-cli/proto_gen/creep.v1.rs`

Both files are copied from `src/creep/`. The plan documents this but there's no tooling to keep them in sync. If the Creep proto evolves, both copies must be updated manually.

**Suggestion:** Add a comment at the top of both copied files pointing to the canonical source, and note the sync step in CLAUDE.md.

### 4. `--addr` not configurable via env var

**File:** `src/creep-cli/src/main.rs:14`

The Creep address defaults to `http://localhost:9090` and can only be overridden via `--addr`. In a containerized setup where Creep might be on a different address, drone hooks would need to be updated to pass `--addr` explicitly. An `CREEP_ADDR` env var fallback would make this more flexible for future environments.

### 5. `format_time` returns raw epoch seconds

**File:** `src/creep-cli/src/main.rs:193`

The human-readable table output shows raw Unix timestamps (`1700000000`) rather than a formatted date. The SKILL.md describes it as "modified timestamp" which is technically correct but unhelpful in practice. Adding `chrono` is overkill, but even a simple `YYYY-MM-DD HH:MM` conversion using basic arithmetic would improve usability. This is a UX issue, not a correctness issue.

### 6. Install script destination inconsistency

**Files:** `src/drones/claude/plugins/install.sh:4` vs `src/drones/claude/base/src/environment.rs:222`

- `install.sh` (for local dev use) installs to `~/.claude/plugins/kerrigan-creep-discovery/`
- `environment::install_plugins()` (for drone container use) installs to `{home}/.claude/plugins/creep-discovery/`

The different names mean a developer running `buck2 run root//src/drones/claude/plugins:install` will get a `kerrigan-creep-discovery` plugin, not `creep-discovery`. If both are installed they'll both work (different plugin names), but the inconsistency is confusing.

## What's Good

- **Graceful fallback throughout** — both `install_plugins` (no-op if `/opt/kerrigan/plugins/creep-discovery` is absent) and the workspace registration hooks (warn + continue if `creep-cli` unavailable) fail safely. This is exactly right for best-effort infrastructure.
- **Proto serde derives** — using `type_attribute(".", "#[derive(serde::Serialize, serde::Deserialize)]")` in `build.rs` is clean; the pre-generated `proto_gen/creep.v1.rs` correctly reflects this.
- **creep-cli's proto_gen is client-only** — the server stubs are correctly omitted (vs `src/creep/proto_gen/creep.v1.rs` which includes both). This is a deliberate, appropriate choice.
- **BUCK file follows existing patterns** — same `OUT_DIR = "proto_gen"` trick used by `src/creep/BUCK`, consistent dep list.
- **SKILL.md is well-written** — clear "when to use / when not to use" structure, correct fallback guidance, accurate command examples.
- **Unit tests for utility functions** — `format_size`, `truncate_hash`, and `format_time` are covered. Minimal but appropriate for pure functions with no I/O.
- **job_run_id validation** — `create_home` already validates the ID (alphanumeric + `-_`), so the path passed to `creep-cli register` is already constrained.
