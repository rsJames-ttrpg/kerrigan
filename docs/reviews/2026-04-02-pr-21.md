## Summary

This PR exposes the hermetic Rust toolchain (`cargo`, `rustc`, `rustdoc`, `rustfmt`,
`clippy-driver`) on PATH after running `./tools/buckstrap.sh`, solving issue #18.

**Approach:** A new `tools:toolchain-bin` Buck2 genrule produces a self-relocatable
wrapper directory. `buckstrap.sh` materialises it and symlinks the wrappers to
`~/.local/bin/`. The wrappers resolve their binary paths at runtime via `readlink -f`
rather than baking in absolute paths.

**Files changed:**

## Findings

### Medium Severity

#### 1. aarch64/Raspberry Pi: `buckstrap.sh` will symlink non-runnable x86_64 binaries

**Location:** `tools/buckstrap.sh` (new toolchain symlink block), `tools/BUCK:toolchain-bin`

The `toolchain-bin` genrule hardcodes `x86_64-linux` source archives:
```python
srcs = [
    "toolchains//:rustc-x86_64-linux",
    "toolchains//:rustfmt-x86_64-linux",
    "toolchains//:clippy-x86_64-linux",
],
```

If `buckstrap.sh` is run on an aarch64 machine (e.g., the Raspberry Pi deployment
target), it will attempt to build the x86_64 genrule and then symlink ELF64/x86-64
binaries into `~/.local/bin/`. These binaries will fail at runtime with "Exec format
error".

The `buckstrap.sh` already has architecture detection for the Buck2 installer itself
(`case "$(uname -m)" in x86_64) ... aarch64) ...`), but the toolchain symlink step
has no such guard.

**Recommended fix:** Add an arch check before the toolchain symlink step:

```bash
if [[ "$(uname -m)" == "x86_64" ]]; then
    echo "Installing hermetic toolchain wrappers to ~/.local/bin/..."
    TOOLCHAIN_BIN=$(buck2 build root//tools:toolchain-bin --show-full-output 2>/dev/null \
      | awk 'NR==1{print $2}')
    ...
else
    echo "  -> NOTE: hermetic toolchain PATH wrappers are x86_64-only; skipping on $(uname -m)"
fi
```

This prevents confusing failures when deploying to the RPi. A proper aarch64 variant
of `toolchain-bin` can be added as a follow-on.

#### 2. `2>/dev/null` swallows build errors; no validation of extracted path

**Location:** `tools/buckstrap.sh`, toolchain symlink block

```bash
TOOLCHAIN_BIN=$(buck2 build root//tools:toolchain-bin --show-full-output 2>/dev/null \
  | awk 'NR==1{print $2}')
if [[ -n "$TOOLCHAIN_BIN" ]]; then
```

The `2>/dev/null` redirects all Buck2 stderr (including build errors) to `/dev/null`.
If the build fails, `$TOOLCHAIN_BIN` will be empty and the script falls through to the
WARNING — but the user has no indication of what went wrong.

Additionally, the `awk` pattern `NR==1{print $2}` doesn't validate that the extracted
value is an actual path (e.g., it could extract a partial line or garbled output).

**Recommended fix:** Remove or narrow the stderr suppression, and add a directory check:

```bash
TOOLCHAIN_BIN=$(buck2 build root//tools:toolchain-bin --show-full-output 2>&1 \
  | grep -E '^root//tools:toolchain-bin ' | awk '{print $2}')
if [[ -d "$TOOLCHAIN_BIN/bin" ]]; then
    ...
else
    echo "  -> WARNING: toolchain-bin output not found at '$TOOLCHAIN_BIN'; skipping"
fi
```

Or at minimum, redirect stderr to a log file rather than discarding it entirely.

### Low Severity

#### 3. Inconsistent HEREDOC quoting in `write_wrapper` — maintenance trap

**Location:** `tools/BUCK`, `toolchain-bin` genrule, `write_wrapper` function

The `write_wrapper` function uses an unquoted heredoc (`<<WRAPPER`) so that `$exec_path`
is interpolated at build time, while `\$` sequences produce literal `$` in the installed
wrapper. The standalone `clippy-driver` block immediately below uses a quoted heredoc
(`<<'WRAPPER'`) with no variable interpolation.

The two styles are intentional but inconsistent, and the reason isn't obvious from the
code. A future maintainer who changes `<<WRAPPER` to `<<'WRAPPER'` (to match the
clippy-driver style) will silently break the wrappers — `$exec_path` won't expand and
all binaries will resolve to `$TOOLCHAIN_ROOT/$exec_path` literally.

**Recommended fix:** Add an inline comment explaining the quoting intent:

```bash
# Note: unquoted <<WRAPPER so $exec_path expands at genrule time.
# \$ sequences become literal $ in the installed script.
cat > "$OUT/bin/$name" <<WRAPPER
```

#### 4. `rustfmt` wrapper only sets `rustc-dist/lib` in `LD_LIBRARY_PATH`, not `rustfmt-dist/lib`

**Location:** `tools/BUCK`, `write_wrapper rustfmt "rustfmt-dist/bin/rustfmt"`

The `rustfmt` wrapper (like the existing `tools:rustfmt` genrule) sets only
`$TOOLCHAIN_ROOT/rustc-dist/lib` in `LD_LIBRARY_PATH`. If the `rustfmt-preview` archive
ever ships additional shared libraries in `rustfmt-dist/lib/`, they won't be found.

This is consistent with the existing `tools:rustfmt` behavior and unlikely to be a
practical issue with current nightly distributions. Worth noting for future maintainers.

#### 5. `toolchain-bin` is built twice in `buckstrap.sh`

**Location:** `tools/buckstrap.sh`

`buck2 build root//...` (existing cache warm-up) already builds `tools:toolchain-bin`.
The subsequent call `buck2 build root//tools:toolchain-bin --show-full-output` rebuilds
(from cache, fast) to retrieve the output path.

This is correct and the second call is practically instant, but it's slightly awkward.
Not a bug; just noting for completeness.

#### 6. `readlink -f` is GNU-specific (macOS incompatible)

**Location:** `tools/BUCK`, wrapper scripts generated by `write_wrapper` and the
clippy-driver heredoc

`readlink -f` requires GNU coreutils and is not available on macOS BSD tools without
installing `coreutils` via Homebrew. This is documented in the spec as a known
limitation. No action needed for this PR — the platform is Linux. Noting it here so
the follow-on issue is clear.

### Positive Aspects

- **Correct self-relocatable design.** Using `readlink -f "$0"` to find `TOOLCHAIN_ROOT`
  at runtime avoids the `uses_local_filesystem_abspaths` limitation of the existing
  `tools:rustfmt` genrule. The output directory can be moved or accessed via symlink
  without breaking.

- **`cp -rfL` is the right call.** Dereferencing symlinks ensures the genrule output is
  self-contained in Buck2's content-addressed store. This prevents dangling symlinks
  inside the archive that would only work on the build machine.

- **`ln -sf` is idempotent.** Re-running `buckstrap.sh` after `buck2 clean` repairs
  symlinks to the new cache location.

- **`set -euo pipefail` in genrule.** Any error during the genrule bash step will
  propagate as a build failure rather than silently producing a broken output.

- **`visibility = ["PUBLIC"]` on `clippy-x86_64-linux` is minimal.** Only the
  target that needs cross-cell visibility gets it — the aarch64 variants are untouched.

- **Test plan is actionable.** The PR description's test plan covers build, smoke-test,
  and self-relocation verification. Easy to follow.

- **Spec and plan documents are well-structured.** The design doc clearly explains why
  Approach A (self-relocatable wrapper directory) was chosen over Approach B (buck2 run
  proxy). The plan's "open questions" section correctly identifies the macOS gap and
  future drone integration work.

---

## Verdict

**Approve with suggestions.** The core implementation is correct and the design is
sound. The aarch64 guard (finding #1) is the most important fix — it prevents confusing
failures on the target deployment hardware. The stderr suppression fix (finding #2)
improves debuggability. The remaining findings are minor style/documentation issues.

The PR can be merged as-is (the aarch64 case doesn't regress existing behavior since
`buckstrap.sh` didn't symlink toolchain binaries before), but the aarch64 guard is a
quick and safe improvement worth making before merge.
